# ADR Format

Architecture Decision Records live in the project's ADR directory — the repo's existing one if it
has ADRs, else `docs/adr/` (the convention). Each is named `ADR-NNNN-slug.md` with sequential
numbering: `ADR-0001-slug.md`, `ADR-0002-slug.md`, etc.

> **Why the `ADR-NNNN` id is load-bearing:** code cites decisions inline by id
> (`// ADR-0006: soft-orphan, not cascade`), and other ADRs reference each other by id. The id is
> the stable anchor that code comments, commits, the registry, and supersession all resolve
> against. The frontmatter `id` and `status` are **required**, not optional.

## Template

```md
---
id: ADR-NNNN
title: {the decision, one line}
status: proposed        # proposed | accepted | deprecated | superseded-by: ADR-MMMM
date: {today — run `date +%Y-%m-%d`}
change: {this change name}     # the link the openspec-archive-change skill uses to promote it
---

# {Short title of the decision}

{1-3 sentences: what's the context, what did we decide, and why.}
```

That's it. An ADR can be a single paragraph. The value is in recording *that* a decision was made
and *why* — not in filling out sections. Everything under the title is free-form prose.

## Required frontmatter

- **`id`** — `ADR-NNNN`, matching the number in the filename. It is the anchor that inline code
  comments and other ADRs cite, so it must be stable and unique.
- **`title`** — the decision in one line. Mirrors the `# ` heading; lets the registry list the
  decision without opening the file.
- **`status`** — one of `proposed | accepted | deprecated | superseded-by: ADR-MMMM`. A reader who
  finds `// ADR-0011` in the code needs the doc to tell them whether it still holds — without a
  status, a stale reference is indistinguishable from a live one.
- **`date`** — `YYYY-MM-DD`, the day it was recorded.
- **`change`** — the name of the change that introduced it. This is the link the `openspec-archive-change`
  skill uses to promote the ADR from `proposed` to `accepted` when the change ships.

**Status lifecycle in this workflow:** new ADRs are born `proposed` (a design decision, not yet
shipped). The `openspec-archive-change` skill promotes every ADR tagged `change: <name>` to `accepted`
when the change is archived. When a later decision replaces one, set the old ADR's status to `superseded-by:
ADR-MMMM` rather than deleting the file — deleting leaves the inline code references and registry
entries dangling.

## Optional sections

Only include these when they add genuine value. Most ADRs won't need them.

- **Context** — the forces in play, when the one-liner under the title isn't enough.
- **Considered options / Alternatives** — only when the rejected alternatives are worth
  remembering. Don't manufacture strawmen to fill the section; if there was one obvious choice,
  leave it out.
- **Consequences** — only when non-obvious downstream effects need to be called out.

## Example

```md
---
id: ADR-0021
title: Self-hosted Docker hosting on a user-managed VPS via Coolify
status: proposed
date: 2026-06-16
change: add-self-hosted-deploy
---

# Self-hosted Docker hosting on a user-managed VPS via Coolify

Production runs on a single user-managed VPS, Docker containers orchestrated by Coolify (app,
Postgres, and the upload sidecar co-located in one private network), deployed by git push. Chosen
over Vercel/PaaS to keep the EU-hosted data boundary and avoid platform lock-in; the cost is that
scaling and failover are manual in v1.
```

## Numbering and dedup — via the registry

The ADR **registry** (`<adr-dir>/README.md`, regenerated by `openspec adr index`) is the canonical
list of what already exists — it is your numbering *and* dedup surface, not a raw directory scan:

- **Number** by taking the highest existing `ADR-NNNN` in the registry and incrementing by one. The
  `id` frontmatter must match the number in the filename.
- **Dedup before minting.** If your decision duplicates an existing ADR, don't write a new one. If
  it revises one, supersede it (new ADR + the old one's `status: superseded-by`), never silently
  duplicate.
- **Regenerate after writing.** Run `openspec adr index` so the registry reflects the new (or
  superseded) ADR. A stale registry is exactly what makes the next session mint a duplicate.

## When to offer an ADR

All three of these must be true:

1. **Hard to reverse** — the cost of changing your mind later is meaningful.
2. **Surprising without context** — a future reader will look at the code and wonder "why on earth
   did they do it this way?"
3. **The result of a real trade-off** — there were genuine alternatives and you picked one for
   specific reasons.

If a decision is easy to reverse, skip it — you'll just reverse it. If it's not surprising, nobody
will wonder why. If there was no real alternative, there's nothing to record beyond "we did the
obvious thing."

### What qualifies

- **Architectural shape.** "We're using a monorepo." "The write model is event-sourced, the read
  model is projected into Postgres."
- **Integration patterns between contexts.** "Ordering and Billing communicate via domain events,
  not synchronous HTTP."
- **Technology choices that carry lock-in.** Database, message bus, auth provider, deployment
  target. Not every library — just the ones that would take a quarter to swap out.
- **Boundary and scope decisions.** "Customer data is owned by the Customer context; other contexts
  reference it by ID only." The explicit no-s are as valuable as the yes-s.
- **Deliberate deviations from the obvious path.** "We're using manual SQL instead of an ORM because
  X." Anything where a reasonable reader would assume the opposite. These stop the next engineer
  from "fixing" something that was deliberate.
- **Constraints not visible in the code.** "We can't use AWS because of compliance requirements."
  "Response times must be under 200ms because of the partner API contract."
- **Rejected alternatives when the rejection is non-obvious.** If you considered GraphQL and picked
  REST for subtle reasons, record it — otherwise someone will suggest GraphQL again in six months.
